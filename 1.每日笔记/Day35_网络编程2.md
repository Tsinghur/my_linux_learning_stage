## Day35——网络编程2

1. accpet的阻塞问题

   accpet取出一个连接后，只要没有新连接来，第二次调用 accept () 照样会阻塞

   - 第一次调用 accept

     - 队列里有连接 → 取走，返回新 fd
     - 队列空 → 阻塞

   - 第二次调用 accept

     **跟第一次完全一样，没有任何区别**：

     - 队列里又有新连接 → 取走，返回新 fd
     - 队列空 → **继续阻塞**

   - 也就是说：

     **accept 每次调用都是独立的，不会因为你之前取过一个就不阻塞。**

2. **recv——网络版的读文件read**

   1. 对端关闭连接 =**类似于文件读到了 EOF（结束）**，一旦读到 EOF：（EOF 是永久状态，不是临时没数据！）

      - **永远就绪**
      - **永远不阻塞**
      - **一直返回 0**

      **对端关闭连接** = TCP 收到 FIN = 数据流到尽头 = **EOF**

   2. 只有一个小区别

      - 普通文件：EOF 是**真的读完了**
      - socket：EOF 代表**对方不再发数据**，但你还能发

3. **read 会阻塞的唯一条件（必须同时满足）**

   1. **当前没有数据可以读**

   2. **写端还开着 / 连接还活着 / 数据流没结束**

      → 内核认为：**以后还会有数据来**

      → 所以**挂起进程，阻塞等待**

4. **两种 “文件”**（网络是流式 I/O，不是普通文件）

   1. **普通文件**：read 永不阻塞
   2. (无缓冲设备文件)**网络 socket / 管道 / 终端 / 设备**：**没数据就阻塞，流关闭就返回 0**

5. **accept中的addrlen字段——==既传入又传出所以传指针，跟bind不同bind仅传入所以传变量==**

   ```c
   struct sockaddr_in client_addr;
   socklen_t addr_len;
   
   // 1. 【传入】先赋值！告诉内核缓冲区大小
   addr_len = sizeof(client_addr);
   
   // 2. 调用 accept
   int client_fd = accept(sockfd, (struct sockaddr *)&client_addr, &addr_len);
   
   // 3. 【传出】现在 addr_len 变成了 实际地址长度
   // 内核已经修改了 addr_len 的值！
   ```

   **为什么长度固定，addrlen 还会被内核修改？**

   因为 **accept / recvfrom / getpeername** 这些函数的设计目标是：**兼容所有协议：IPv4、IPv6、UNIX 域套接字**

   它们的长度**不一样**：

   - IPv4：`sizeof(struct sockaddr_in)` = **16**
   - IPv6：`sizeof(struct sockaddr_in6)` = **28**
   - UNIX 域：`sizeof(struct sockaddr_un)` = **110**

   内核需要**统一接口**，所以必须：

   - 你**传入**你给的缓冲区最大长度
   - 内核**传出**实际用了多少

6. ==**select监听到读就绪意味着什么**==

   **socket_fd 读就绪 = 内核告诉你：这个 socket 的接收缓冲区里已经有数据了，现在调用 recv /read 一定不会阻塞，能直接读到东西。**

   分场景说清楚（你写 select 服务器必用）

   1. 如果是 **监听 socket（listen_fd）**

   读就绪 = **有客户端完成三次握手**

   全连接队列里已经有连接了

   → 现在调用 `accept()` 不会阻塞，能直接拿到 client_fd

   2. 如果是 **已连接 socket（client_fd）**

   读就绪 = **接收缓冲区里有数据**

   下面任意一种情况都会触发读就绪：

   - 客户端发了数据，内核收到放进缓冲区
   - 客户端关闭连接，发了 FIN 包
   - 连接出现异常（RST）

   只要 select 告诉你它就绪了：

   - `recv()` 一定**不阻塞**
   - 要么读到数据（返回 > 0）
   - 要么读到 EOF（返回 = 0，对端关闭）
   - 要么读到错误（返回 < 0）

   3. 一个极易混淆的关键点

   **读就绪 ≠ 一定能读到有效数据**

   它只保证：**调用 recv 不会阻塞**

   比如：

   - 对端关闭 socket → 缓冲区 EOF
   - 内核会让 fd 读就绪
   - 但你 `recv` 会返回 **0**

   4. 和你之前知识串起来

   - socket 内部有 **接收缓冲区**
   - 缓冲区空 + 连接活着 → **未就绪，select 会等**
   - 缓冲区有数据 / 对端关闭 → **变为就绪**
   - select 发现就绪 → 唤醒进程，告诉你可以读了

   **终极总结**

   **socket_fd 读就绪 = 可以安全调用 recv/accept，不会阻塞。**

   - 监听 fd 就绪 → 能 accept
   - 客户端 fd 就绪 → 能 recv

7. **对端关闭后**，继续两次终端输入然后将其send会导致进程异常终止

   1. 你**第一次 send**：==内核发现对端关闭==，**悄悄发 RST 重置连接**，但 **send 本身不会报错！不会触发 SIGPIPE！**，**第一次回车：程序正常跑完循环！**
   2. **第二次send：大概率触发 SIGPIPE，进程直接退出**

8. **==关于select的阻塞问题==**

   1. **客户端断开连接**

      主要fd还在监听集合中，且你一直不read/recv，那么select就会永远返回“读就绪”，形成死循环，**CPU跑满**

      因为：**对端关闭连接，在 TCP 里就是一个 “读事件”**

   2. **客户端连接时**

      - 没有发送数据，select正常阻塞
      - 若是发送数据，fd还在监听集合中，且一直不读，则**仍然会死循环**

   ```c
   fd 在监听集合里，一直不调用 recv
   ├── 接收缓冲区 有数据
   │    → select 立刻返回可读
   │    → 循环不阻塞 → CPU 跑满
   ├── 接收缓冲区 无数据，但对端已关闭（FIN/RST）
   │    → select 立刻返回可读
   │    → 循环不阻塞 → CPU 跑满
   └── 接收缓冲区 空，连接正常活着
        → select 阻塞等待
        → 不循环，CPU 休息
   ```
